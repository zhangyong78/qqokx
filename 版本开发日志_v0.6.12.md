# 版本开发日志 v0.6.12

更新时间：`2026-06-06`

## 这轮做了什么

这一轮主要围绕“轮询版实盘策略如何在小机器上更稳地扩容”做了三件事：

1. 调整默认轮询参数，把默认 `poll_seconds` 从 `3s` 提高到 `10s`
2. 增加共享行情层，让同一进程里相同 `instId + bar` 的 K 线读取复用
3. 把触发价和订单状态尽量切到 WebSocket 缓存优先，读不到时再回退到 REST

目标不是重写成完整事件驱动架构，而是在尽量少改现有策略逻辑的前提下，把 `2核 2G` 服务器跑 `2 套 API / 20 个左右 session` 这类场景先稳住，并给后续扩到 `5 套 API` 打基础。

## 主要功能更新

### 1. 默认轮询间隔调整

默认值调整为：

- [okx_quant/models.py](/D:/qqokx/okx_quant/models.py)
  - `StrategyConfig.poll_seconds = 10.0`
- [okx_quant/position_protection.py](/D:/qqokx/okx_quant/position_protection.py)
  - `OptionProtectionConfig.poll_seconds = 10.0`
- [okx_quant/backtest_ui.py](/D:/qqokx/okx_quant/backtest_ui.py)
  - 配置反序列化回退值同步改为 `10.0`

这次调整的考虑是：

- 当前主实盘场景以 `1H / 4H` 为主，`3s` 对中低频策略过密
- 小机器上更容易先撞到的是读请求密度、线程数和网络抖动，而不是纯 CPU
- 先把默认值调稳，比继续维持高频轮询更符合当前实盘使用方式

### 2. 共享 K 线读取层

新增文件：

- [okx_quant/market_data_hub.py](/D:/qqokx/okx_quant/market_data_hub.py)

核心能力：

- 按 `instId + bar` 聚合同一进程里的 candle 读取
- 同币种同周期只保留一个共享 feed
- 多个策略会话优先读共享缓存，拿不到时自动回退原来的 REST 读取
- 刷新节奏按周期自适应，而不是每个 session 都独立硬轮询

接入点：

- [okx_quant/engine.py](/D:/qqokx/okx_quant/engine.py)
  - `StrategyEngine` 支持注入 `market_data_hub`
  - `_get_candles_with_retry()` 优先走共享 hub
- [okx_quant/ui_shell.py](/D:/qqokx/okx_quant/ui_shell.py)
  - 应用启动时创建全局共享 hub
  - 应用关闭时统一停止 hub
- [okx_quant/ui_strategy_sessions.py](/D:/qqokx/okx_quant/ui_strategy_sessions.py)
  - 各类 `StrategyEngine(...)` 创建点统一透传共享 hub

这一步的主要收益是：

- 相同币种 / 相同周期 / 不同 API / 不同多空方向的重复 K 线读取会明显减少
- 对 `5 币种 + 1H + 多 API` 这类场景，K 线 REST 请求量通常能降掉大头

### 3. 触发价优先公共 WS

本轮继续把触发价读取接到了公共行情缓存：

- [okx_quant/okx_client.py](/D:/qqokx/okx_quant/okx_client.py)
  - `get_trigger_price(..., environment=...)` 优先读取公共 WS ticker
  - 取不到缓存时，回退到原有 REST ticker / mark-price
- [okx_quant/engine_retry_policy.py](/D:/qqokx/okx_quant/engine_retry_policy.py)
  - 统一封装触发价读取优先级
- [okx_quant/engine.py](/D:/qqokx/okx_quant/engine.py)
  - 实盘各触发价调用点继续向下透传 `environment`
- [okx_quant/position_protection.py](/D:/qqokx/okx_quant/position_protection.py)
  - 持仓保护线程的触发价读取也兼容环境参数

这样做以后：

- 活跃持仓、本地保护、动态止损等链路更容易命中公共 ticker 缓存
- 触发价读取不再每次都依赖单独 REST 查询

### 4. 订单状态优先私有 WS

订单状态读取这轮也做了轻量增强：

- [okx_quant/engine_retry_policy.py](/D:/qqokx/okx_quant/engine_retry_policy.py)
  - `get_order(...)` 先查私有 WS 缓存
  - 如果缓存已是终态，直接返回
  - 如果还是旧的 `live / partially_filled`，会短等一次私有 WS 新更新
  - 仍拿不到时才回退到原有 REST `get_order(...)`

这一层的目标是：

- 已成交 / 已撤单这类终态尽量不再走 REST
- 避免活跃订单状态轮询过于依赖 HTTP
- 又不因为只看缓存而把订单卡在旧状态

## 当前结论

这轮不是“大改架构”，而是一次偏务实的“实盘扩容减压”版本：

- 先把默认轮询调稳
- 再把最值钱的公共数据复用做掉
- 再把最频繁的价格 / 订单状态读取切到 WS 优先

对当前项目的意义是：

- 轮询版策略还可以继续跑，但不再是每个 session 都完全各拉各的
- `2 套 API + 1H + 20 个左右 session` 在 `2核 2G` 上已经更适合作为起步规模
- 后续如果继续往 `5 套 API` 扩，当前这层优化能先扛住一部分压力

## 后续建议

后面如果继续沿这个方向迭代，建议优先级如下：

1. 先继续观察 `2 套 API / 20 session` 的实盘稳定性
2. 再灰度扩到 `5 套 API`
3. 如果后面要长期跑更大规模，再考虑把更多交易后状态和行情读取改成事件驱动
4. 如果未来要上更多周期或更多策略组合，再评估是否需要进一步拆分进程或升级服务器规格
