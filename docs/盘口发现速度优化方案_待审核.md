# 盘口发现速度优化方案（待审核）

## 1. 背景判断

当前这个小工具的主目标，不是全市场量化扫描，而是：

- 尽快看到目标双腿盘口
- 尽快判断当前可执行价差
- 尽快在几个候选合约之间切换

也就是说，产品定位应该更像“交易报价台”，而不是“全市场扫描器”。

现在体感慢，核心不是单点网络慢，而是整体交互模式还偏“周期扫描 + 全量刷新”。

## 2. 当前慢的根因

结合代码，当前主要有 4 个瓶颈。

### 2.1 默认工作流仍然偏“全市场扫描”

在 [okx_quant/arbitrage_ui.py](/D:/qqokx/okx_quant/arbitrage_ui.py:2568) 里，界面会按固定周期自动刷新扫描；扫描动作会走 [okx_quant/arbitrage_ui.py](/D:/qqokx/okx_quant/arbitrage_ui.py:2739) 的 `scan_opportunities()`。

而扫描本身在 [okx_quant/arbitrage/arbitrage_scanner.py](/D:/qqokx/okx_quant/arbitrage/arbitrage_scanner.py:226) 会做这些事：

- 拉现货合约列表
- 拉永续合约列表
- 拉交割合约列表
- 分别拉 `SPOT / SWAP / FUTURES` 三类 ticker
- 再拼接出全部机会列表

这套逻辑适合“找机会”，不适合“迅速看盘口”。

### 2.2 盘口区域虽然有 WS，但仍然是定时线程拉取模型

在 [okx_quant/arbitrage_ui.py](/D:/qqokx/okx_quant/arbitrage_ui.py:2043) 到 [okx_quant/arbitrage_ui.py](/D:/qqokx/okx_quant/arbitrage_ui.py:2319) 之间，盘口面板仍然是：

- 定时器触发
- 起线程
- 再去取快照
- 再整体回填 UI

这意味着它本质上还是“轮询式 UI”，不是“推送式 UI”。

### 2.3 WS 只要没凑齐四份缓存，就会回退到 REST

在 [okx_quant/arbitrage_ui.py](/D:/qqokx/okx_quant/arbitrage_ui.py:2168) 到 [okx_quant/arbitrage_ui.py](/D:/qqokx/okx_quant/arbitrage_ui.py:2209) 之间，只有下面四份数据都到齐，才算使用 WS：

- 左腿 ticker
- 右腿 ticker
- 左腿 order book
- 右腿 order book

只要其中任意一份没到，就会回退 REST：

- `get_ticker()` 两次
- `get_order_book()` 两次

对应接口在 [okx_quant/okx_client.py](/D:/qqokx/okx_quant/okx_client.py:906) 和 [okx_quant/okx_client.py](/D:/qqokx/okx_quant/okx_client.py:913)。

这会导致：

- 初次切换合约时慢一拍
- WS 短暂抖动时又退回 REST
- UI“明明是实时盘口，实际却偶发走 HTTP”

### 2.4 每次刷新都在全量清空并重绘盘口表格

在 [okx_quant/arbitrage_ui.py](/D:/qqokx/okx_quant/arbitrage_ui.py:2368) 里，盘口表每次刷新都会：

- 删除所有行
- 再重新插入所有行

这会造成：

- UI 视觉抖动
- Tk 渲染开销累积
- 用户感觉价格“跳”和“迟”

## 3. 优化方向建议

建议把这块明确改成“盘口优先”的产品路线，而不是继续在现有扫描模式上微调。

### 方向 A：产品层改成“报价台优先”

默认首页不再强调“全市场扫描”，而是强调：

- 当前关注币种
- 当前腿 / 目标腿
- 买一卖一
- 可执行价差
- 最近切换列表

建议把使用路径改成：

1. 先选币种
2. 立即展示默认双腿盘口
3. 上下切换交割合约
4. 直接看到可执行价差变化

也就是“先看盘，再决定是否移仓/开仓”，而不是“先扫描，再进盘口”。

### 方向 B：数据层改成“WS 驱动、REST 兜底”

建议建立一个轻量的“行情快照中心”，核心原则是：

- 当前选中的双腿，始终常驻订阅
- 候选合约列表，按 watchlist 预订阅
- UI 只消费最新缓存，不主动周期拉 REST
- REST 仅用于冷启动兜底，不参与常规刷新

重点改法：

1. 盘口 UI 改成事件驱动
- WS 一到数据就推 UI 更新
- 不再每 250ms 起一个刷新线程

2. 降低 WS 判定门槛
- 不要求 4 份缓存同时到齐才展示
- ticker 先到就先显示买一卖一
- order book 晚到再补深度
- UI 上显示“部分实时 / 深度补齐中”

3. 增加数据时效标记
- 每条行情带 `received_at`
- 超过阈值就标“陈旧”
- 陈旧时才触发单次 REST 补拉

这样能把“是否能快速看到价格”的判断，从“整包是否齐全”改成“最关键价格是否已到”。

### 方向 C：界面层改成“BBO 优先，深度其次”

用户真正第一时间需要的不是 10 档完整盘口，而是：

- 买一
- 卖一
- 中间价
- 当前可执行价差
- 变化方向

建议盘口区拆成两层：

1. 顶部极速报价条
- 左腿买一/卖一
- 右腿买一/卖一
- 价差绝对值
- 价差百分比
- 最近一次更新时间

2. 下方深度区
- 只展示 5 档
- 仅在用户需要时展开 10 档

这样用户会先“看到价格”，再决定要不要看深度。

### 方向 D：切换逻辑改成“最近候选 + 预热”

对交割移仓这个场景，最常见动作不是扫全市场，而是：

- 同一币种下切换近周 / 次周 / 当季 / 次季

建议加一条“目标腿快速切换带”：

- 当前合约
- 近周
- 次周
- 当季
- 次季

这些候选合约在用户选中当前持仓后，立即一起预订阅 ticker + books5。

这样用户切换目标腿时，不需要再等二次预热。

## 4. 具体实施方案

### 第一阶段：快速见效版

目标：先把“看价格慢”解决到可用。

改动范围：

- 默认关闭或弱化全市场自动扫描
- 新增“盘口快看模式”为默认入口
- 当前选中双腿改为常驻 WS 订阅
- WS 数据部分到齐时就先显示 BBO
- 盘口表格改为增量更新，不再全删全插
- 交割移仓页增加“候选交割合约快速切换”

预期效果：

- 首次切腿后 300ms 到 800ms 内看到买一卖一
- 稳态刷新接近 WS 实时
- 视觉抖动明显减少

### 第二阶段：结构优化版

目标：把这块从“能用”提升到“顺手”。

改动范围：

- 抽出 `MarketQuoteHub` 或类似行情中心
- UI 不直接调 client 拉数据，只订阅快照
- watchlist 机制持久化
- 引入“最近使用合约组”
- 增加“价差跳动高亮”和“时效状态”

预期效果：

- 页面切换更流畅
- 多 tab 共用行情缓存
- 后续扩展自动移仓、自动平仓也更稳

### 第三阶段：体验强化版

目标：把它做成真正的“手工交易辅助台”。

可选增强：

- 自定义快捷键切换目标腿
- 价差阈值颜色提示
- 单币种固定工作区
- 一键对比多张交割合约
- 小窗悬浮报价条

这阶段不是必须，但很适合这个工具的真实定位。

## 5. 我建议的优先级

如果只做一轮优化，我建议按这个顺序：

1. 去掉“盘口显示依赖完整 WS 四件套”的限制
2. 改成 BBO 优先显示
3. 盘口表格改为增量刷新
4. 交割页做候选合约预热订阅
5. 把全市场自动扫描降级为辅助功能

原因很简单：

- 这 5 项最直接影响“看到价格快不快”
- 对现有架构侵入还可控
- 比重写整套更容易快速验收

## 6. 验收指标建议

建议审核时，不只看“主观感觉”，还要看 4 个客观指标。

### 6.1 冷启动指标

- 选中一组双腿后，`<= 1s` 出现首屏买一卖一

### 6.2 稳态指标

- WS 正常时，价格刷新间隔体感接近实时
- 不再出现频繁回退 REST 的状态跳变

### 6.3 切换指标

- 同币种切换目标交割合约时，`<= 300ms` 更新主要报价

### 6.4 UI 指标

- 盘口区域不闪烁
- 不整表重绘
- 状态栏能明确看到“实时 / 补齐中 / 陈旧”

## 7. 结论

这块不建议继续沿着“扫描更快一点、定时器更勤一点”的方向修修补补。

更正确的方向是：

- 产品上，从“扫描器”转成“报价台”
- 数据上，从“轮询取快照”转成“WS 推送快照”
- 界面上，从“10 档全量表格”转成“BBO 优先 + 深度补充”
