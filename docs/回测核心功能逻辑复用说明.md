# 回测核心功能逻辑复用说明

本文总结本项目回测模块中值得迁移到股票程序化交易项目的重点功能和实现逻辑，重点覆盖 UI K 线图、开平仓信号、止损止盈、回测报告和审计导出。

## 1. 核心文件地图

| 文件 | 作用 |
| --- | --- |
| `okx_quant/backtest.py` | 回测主引擎。负责加载 K 线、执行策略、逐 K 线撮合开平仓、计算止盈止损、生成资金曲线和统计报告。 |
| `okx_quant/backtest_ui.py` | 回测 UI。负责参数输入、历史快照、交易明细表、K 线图/净值/回撤图绘制、缩放拖动和报告展示。 |
| `okx_quant/backtest_export.py` | 回测报告导出。负责单次/批量报告文本、参数摘要、交易明细和重点结论。 |
| `okx_quant/backtest_audit.py` | 审计文件导出。负责资金曲线 CSV、操作日志 CSV、审计 JSON 清单。 |
| `okx_quant/models.py` | 通用数据模型。包含 `Candle`、`StrategyConfig`、`SignalDecision`、`OrderPlan`、`DynamicProtectionRule` 等。 |
| `okx_quant/strategy_runtime_registry.py` | 策略运行时分类。回测引擎根据策略 family 分发到不同执行逻辑。 |
| `okx_quant/strategy_ui_schema.py` | 策略 UI 可见性和参数差异。用于同一套回测 UI 适配多种策略。 |

## 2. 数据结构设计

本项目做得好的地方，是先把回测结果沉淀成稳定的数据模型，再让 UI、图表、报告、审计共用这些模型，而不是各处临时拼字段。

### 2.1 输入层

- `Candle`：标准 OHLCV K 线，字段为 `ts/open/high/low/close/volume/confirmed`。
- `Instrument`：交易品种规则，字段包括最小变动价位、最小下单数量、合约信息等。
- `StrategyConfig`：完整策略配置，包含品种、周期、均线、ATR、止盈方式、资金管理、手续费、滑点、多周期过滤、日线过滤等。
- `SignalDecision`：策略信号输出，包含方向、原因、信号 K 线时间、入场参考价、ATR、均线值等。
- `OrderPlan`：把信号变成可执行计划，包含方向、数量、入场价、止损、止盈、ATR 和触发信息。

### 2.2 结果层

- `BacktestTrade`：单笔已平仓交易。保存开仓/平仓索引、时间、价格、初始止损、初始止盈、数量、手续费、滑点、资金费、净盈亏、R 倍数和平仓原因。
- `BacktestReport`：汇总统计。包含交易数、胜率、总盈亏、平均盈亏、盈亏比、Profit Factor、平均 R、最大回撤、收益率、手续费、滑点成本、资金费等。
- `BacktestResult`：完整结果容器。包含 K 线、交易列表、报告、均线/ATR 序列、净值曲线、回撤曲线、月度/年度统计、开放仓位和策略参数快照。

迁移建议：股票项目也保持这三层结构：`Trade` 记录单笔，`Report` 记录汇总，`Result` 作为 UI/导出统一数据源。

## 3. 回测主流程

入口是 `run_backtest()`，主流程如下：

1. 校验回测参数：K 线数量不能为负，不能超过 `MAX_BACKTEST_CANDLES`，开始时间不能晚于结束时间。
2. 根据策略需要预加载 K 线，预留指标计算 warm-up 区间，避免均线/ATR 刚开始不稳定。
3. 加载交易品种规则，用于价格 tick、数量精度、最小数量校验。
4. 如果策略需要多周期过滤，额外加载高周期 K 线并生成方向过滤序列。
5. 如果启用日线过滤，聚合/加载日线并生成方向过滤序列。
6. 进入 `_run_backtest_with_loaded_data()`，按策略 family 分发到对应回测函数。
7. 回测函数逐根 K 线执行：先处理已有仓位退出，再判断新信号和新订单。
8. 生成 `BacktestTrade` 列表后，统一计算均线、ATR、资金曲线、净值曲线、回撤曲线、月度/年度统计和 `BacktestReport`。

关键思想：策略只负责“什么时候做什么方向”，回测引擎负责“按交易规则如何成交、如何止损止盈、如何计费、如何统计”。

## 4. 开仓信号与订单计划

当前项目支持多类策略，但共同抽象是：

1. 策略函数输出 `SignalDecision`。
2. 回测引擎用 `_build_backtest_order_plan()` 或对应策略内的计划构建逻辑，把信号变成 `OrderPlan`。
3. `OrderPlan` 固定包含：
   - `signal`：long/short。
   - `entry_reference`：入场参考价。
   - `stop_loss`：止损价。
   - `take_profit`：止盈价或动态止盈触发价。
   - `size`：按资金管理规则计算出的数量。
   - `atr_value`：用于复盘、报告和止损止盈解释。

资金管理支持三种模式：

- 固定风险金额：每笔亏到初始止损时亏固定金额。
- 固定数量：直接按指定数量下单。
- 风险百分比：按当前权益的一定百分比做风险金额，可选择复利。

迁移到股票项目时，建议保留 `SignalDecision -> OrderPlan -> OpenPosition -> Trade` 这条链路。这样策略模块和交易撮合模块边界清楚，后续接实盘也更容易。

## 5. 止损止盈逻辑

### 5.1 固定止损止盈

固定模式下，回测引擎保存入场时的初始止损和止盈。平仓判断逻辑：

- 多头：`low <= stop_loss` 触发止损，`high >= take_profit` 触发止盈。
- 空头：`high >= stop_loss` 触发止损，`low <= take_profit` 触发止盈。
- 非动态模式下，如果同一根 K 同时触发止损止盈，当前普通逻辑偏保守，优先止损。

### 5.2 单根 K 线内路径判断

动态止盈模式做得更细。它不是只看 high/low，而是把一根 K 线近似成价格路径：

- 阳线：`open -> low -> high -> close`
- 阴线：`open -> high -> low -> close`

然后逐段判断价格先碰到哪个价位。这样可以更合理地处理“同一根 K 内先到止盈触发价，再回落打到移动止损”的情况。

相关函数：

- `_candle_path_points()`：普通 K 线路径。
- `_same_candle_path_points()`：开仓同 K 后的路径。
- `_first_touched_exit_on_segment()`：在线段内判断先碰到止损还是止盈。
- `_process_dynamic_position_segment()`：动态止盈场景下逐段更新止损并判断退出。

迁移建议：股票分钟线或日线回测都可以使用这种路径近似。若做日线股票回测，要额外处理涨跌停、开盘一字板、集合竞价和成交量不足。

### 5.3 动态止盈/保本/锁盈

动态止盈不是直接平仓，而是把“下一触发 R”临时作为 `take_profit` 触发点。一旦价格达到该 R 倍数：

1. 根据规则把 `stop_loss` 上移或下移。
2. 更新下一次触发 R。
3. 继续持仓，直到价格反向打到新的 `stop_loss`。

核心公式：

- `risk_per_unit = abs(entry_price - initial_stop_loss)`
- 多头触发价：`entry_price + risk_per_unit * trigger_r + fee_offset`
- 空头触发价：`entry_price - risk_per_unit * trigger_r - fee_offset`
- 保本价会加入手续费偏移，避免“账面保本但扣费后亏损”。

动态规则由 `DynamicProtectionRule` 表达：

- `trigger_r`：达到多少 R 时触发。
- `action`：`break_even` 或 `lock_profit`。
- `lock_r`：锁定多少 R。
- `trail_mode`：是否阶梯移动。
- `trail_every_r` / `trail_add_r`：每前进多少 R，再额外锁多少 R。

示例规则：

- 2R 保本。
- 5R 锁 4R。
- 5R 后每多走 1R，止损也顺移 1R。

相关函数：

- `_dynamic_trigger_price()`：计算下一触发价。
- `_dynamic_break_even_price()`：计算含手续费偏移的保本价。
- `_dynamic_stop_price()`：计算锁 R 后的止损价。
- `_advance_dynamic_stop()`：价格顺向发展时推进止损。
- `_dynamic_stop_exit_reason()`：把退出原因标记为止损、保本、锁定 N R 等。

迁移建议：股票项目可以把 `funding_rate` 去掉，但保留手续费和滑点偏移。A 股还要考虑印花税、最低佣金、过户费等费用模型。

## 6. 手续费、滑点和资金成本

当前交易计算使用净盈亏：

`pnl = gross_pnl - entry_fee - exit_fee - funding_cost`

同时单独记录：

- `entry_fee`
- `exit_fee`
- `total_fee`
- `slippage_cost`
- `funding_cost`
- `gross_pnl`
- `pnl`

滑点逻辑：

- 多头开仓价格上调，平仓价格下调。
- 空头开仓价格下调，平仓价格上调。
- 所有价格最终按 tick size 对齐。

迁移到股票时建议：

- A 股：佣金、印花税、过户费、最低收费、T+1 卖出限制、涨跌停无法成交。
- 港股/美股：佣金、平台费、SEC/TAF、融资利息、盘前盘后流动性。
- 如果只做多头股票，可删除空头逻辑，但建议保留方向字段，后续支持融券或期货会更方便。

## 7. UI K 线图逻辑

图表在 `backtest_ui.py` 的 `_draw_chart()` 中完成。它不是只画价格，而是一个复合复盘图：

1. 上方面板：K 线、均线、开仓点、平仓点、交易连线、止损线、止盈线、开放仓位。
2. 中间面板：净值曲线。
3. 下方面板：回撤百分比曲线。

图表视口由 `_ChartViewport` 管理：

- `start_index`：当前显示起点。
- `visible_count`：当前显示 K 线数量。
- 鼠标滚轮缩放。
- 左键拖动平移。
- 大窗查看复盘。
- hover 显示当前 K 线、均线、ATR、净值等信息。

性能优化：

- 默认只显示最近约 900 根 K。
- K 线太多时使用 `render_stride` 跳采样。
- 拖动/缩放时先 fast render，再延迟完整渲染。

交易标注方式：

- `entry_index` 和 `exit_index` 对齐 K 线横坐标。
- 开仓点用方向颜色圆点。
- 平仓点用绿色/红色圆点。
- 开仓价到平仓价画连线。
- 初始止损、初始止盈在持仓区间画虚线。
- 平仓标签显示止盈、止损、保本、锁定 R 等。

迁移建议：股票项目可以直接沿用这个 UI 思路。即使换成 Web 前端，也建议保留同样的数据接口：前端只需要 `candles + trades + indicators + equity_curve + drawdown_curve`。

## 8. 回测报告逻辑

报告由 `format_backtest_report()` 和 `backtest_export.py` 共同负责。

### 8.1 核心统计

`BacktestReport` 计算：

- 总交易数、盈利笔数、亏损笔数、保本笔数。
- 胜率。
- 总盈亏、平均盈亏。
- 毛利润、毛亏损。
- Profit Factor。
- 平均盈利、平均亏损、盈亏比。
- 平均 R 倍数。
- 最大回撤和最大回撤百分比。
- 止盈触发次数、止损触发次数。
- 期末权益、总收益率。
- Maker/Taker 手续费、总手续费。
- 滑点成本、资金费成本。
- 开放仓位和人工托管仓位信息。

### 8.2 资金曲线和回撤

资金曲线按已平仓交易计算：

1. 每笔交易的 `pnl` 归入 `exit_index` 对应 K 线。
2. 按 K 线累加得到 `equity_curve`。
3. `net_value_curve = initial_capital + equity_curve`。
4. 用净值高点回撤计算 `drawdown_curve` 和 `drawdown_pct_curve`。

月度/年度统计按交易平仓时间分组，计算每期交易数、胜率、总盈亏、收益率、期初/期末权益和期内最大回撤。

### 8.3 导出内容

单次回测导出：

- 主报告 txt。
- 资金审计 CSV：每根 K 的实现权益、浮动权益、回撤等。
- 操作日志 CSV：开仓、平仓、移交、快照等事件。
- 审计 JSON：配置、区间、文件清单、统计摘要和方法说明。

批量回测导出：

- 根报告：矩阵对比。
- 明细目录：每个参数组合一份完整单次报告。
- 批量审计 JSON：记录所有明细文件和结果摘要。

迁移建议：不要只导出一个漂亮报告。保留 CSV 和 JSON 审计文件，方便后续自动对账、复盘、回归测试和给其他系统读取。

## 9. 可复用架构建议

股票程序化交易项目可以按下面结构复用：

```text

  candle_loader.py       # 股票 K 线、复权、交易日历、停牌处理
core/
  models.py              # Candle / Instrument / StrategyConfig / Trade / Report / Result
  indicators.py          # MA / EMA / ATR / 自定义指标
  backtest_engine.py     # run_backtest、逐 K 线撮合、止损止盈

  test_exit_order.py

```

## 10. 股票项目迁移注意事项

股票和数字货币/合约不同，迁移时重点改这些地方：

1. 交易时间：使用交易所日历，跳过周末、节假日、午休、停牌。
2. 复权：明确使用前复权、后复权还是不复权；信号和成交价格必须一致。
3. 涨跌停：价格达到止损/止盈也不一定能成交，需要模拟不可成交或排队成交。
4. T+1：A 股当日买入不能当日卖出，回测撮合层要限制卖出时间。
5. 手续费：佣金、印花税、过户费、最低佣金要分开算。
6. 滑点：股票滑点最好和成交额、成交量、盘口流动性挂钩。
7. 成交量约束：不能假设无限流动性，单笔成交量应限制为当根成交量的一定比例。
8. 分红配股：长期回测必须处理公司行动，否则收益曲线会失真。
9. 多标的组合：股票项目通常需要组合级资金、仓位上限、行业暴露、单票最大持仓。
10. 风险控制：增加最大持仓天数、最大组合回撤停机、连续亏损降仓等组合层规则。

## 11. 最小迁移清单

如果只想先把核心能力搬到另一个股票项目，优先迁移这些能力：

1. `Candle / StrategyConfig / SignalDecision / OrderPlan / BacktestTrade / BacktestReport / BacktestResult` 数据模型。
2. `run_backtest()` 的主流程：加载数据、预热、策略分发、生成结果。
3. `_create_open_position()`、`_try_close_position()`、`_build_closed_trade()`。
4. 动态保护规则：`DynamicProtectionRule`、保本、锁 R、阶梯移动止损。
5. 资金曲线和回撤：`_build_equity_curve()`、`_build_drawdown_curves()`、`_build_period_stats()`。
6. K 线图协议：`candles + trades + indicators + net_value_curve + drawdown_curve`。
7. 报告导出：主报告、交易明细、资金审计 CSV、操作日志 CSV、审计 JSON。

## 12. 建议保留的测试

迁移后一定要补这些测试：

- 同一根 K 同时触发止损/止盈时，退出顺序符合规则。
- 动态止盈到 2R、5R、阶梯移动后，止损价移动正确。
- 手续费和滑点计入后，净盈亏、R 倍数和保本价正确。
- 资金曲线只在平仓 K 线确认收益，不提前确认。
- 多周期/日线过滤不能使用未来数据。
- 股票涨跌停、T+1、停牌、成交量不足时，成交逻辑正确。

这套回测模块最值得复用的不是某一个策略，而是“策略信号、订单计划、持仓状态、交易结果、统计报告、图表复盘”分层清楚。其它项目只要保持这个结构，后面扩策略、换 UI、接实盘、做批量参数测试都会更稳。
