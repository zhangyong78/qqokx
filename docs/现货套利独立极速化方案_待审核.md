# 现货套利独立极速化方案（待审核）

## 1. 我对需求的理解

这次目标不是把工作台里的一个 tab 再优化一点，而是把“现货套利”当成一个独立的临战工具来设计。

它的真实使用场景是：

- 服务器继续跑量化主系统
- 本地人工打开“现货套利”
- 锁定当前交割合约和目标交割合约
- 后续只关注这 2 到 4 个真实要交易的标的
- 用最快速度看盘口、判断价差、完成移仓或平仓

所以它更像：

- 独立交易终端

而不是：

- 量化工作台里的一个附属窗口

这个理解下，优化方向就不能再停留在“减少几个刷新动作”，而要直接改成“独立运行时 + 最小化路径 + 专用资源”。

## 2. 当前结构为什么不适合这个目标

现在的“现货套利”从主工作台内打开，本质上仍是一个共享环境里的工具窗口。

可以看到：

- 主程序入口仍然是整套桌面应用，[main.py](/D:/qqokx/main.py:22)
- “现货套利”由主工作台菜单拉起，[okx_quant/ui_shell.py](/D:/qqokx/okx_quant/ui_shell.py:4057)
- 打开时只是新建一个 `ArbitrageWindow`，但继续复用主进程里的 `root` 和 `client`，[okx_quant/ui_backtest_entry.py](/D:/qqokx/okx_quant/ui_backtest_entry.py:98)

这意味着它天然存在几个问题。

### 2.1 它和主工作台抢同一个进程

- 共用 Tk 主线程
- 共用同一个 Python 进程
- 共用同一个 `OkxRestClient`
- 共用同一套日志、状态、定时任务氛围

对“临战快交易”来说，这种共享不是优势，而是负担。

### 2.2 它保留了扫描器式思维

即使用户已经锁定合约，工具里仍然保留：

- 全市场扫描
- 多 tab 切换
- 泛化的持仓/预览/图表刷新

而你真正要的，是：

- 锁定后只看极少数腿
- 资源只服务当前这一笔交易

### 2.3 执行链里还有很多“通用型动作”

例如移仓路径 [okx_quant/arbitrage/arbitrage_executor.py](/D:/qqokx/okx_quant/arbitrage/arbitrage_executor.py:1405) 仍然兼顾：

- 从账本查 entry
- 从请求补 entry
- 通用分批
- 通用挂单追单
- 完成后再回写本地账本

这些逻辑没错，但它是“通用可靠性优先”，不是“专用极限性能优先”。

### 2.4 本地状态持久化还是同步整表读写

本地套利账本目前是整份读出、修改、整份写回，[okx_quant/arbitrage/position_ledger.py](/D:/qqokx/okx_quant/arbitrage/position_ledger.py:27)。

这对普通工具没问题，但如果要做临战终端，应该改成：

- 内存主状态
- 异步落盘
- 交易后再持久化

而不是每次关键动作都走文件读写路径。

## 3. 目标架构

我建议直接把它重构成：

- 主工作台保留入口
- 现货套利以独立进程运行
- 进入后切到“锁定合约模式”
- 后台只维护当前套利上下文需要的极少数数据流

一句话总结：

- `入口仍在主程序`
- `运行时完全独立`
- `锁定后只服务当前合约组`

## 4. 推荐方案

## 方案结论

我推荐做成：

- `Arbitrage Fast App` 独立专用进程

主工作台里只保留一个入口按钮：

- 打开现货套利极速版

点击后不是再开一个共享窗口，而是直接启动一个独立子进程。

## 5. 独立进程方案

### 5.1 启动方式

建议新增一个独立入口，例如：

- `python main.py --app arbitrage-fast`

或者单独可打包为：

- `qqokx_arbitrage_fast.exe`

主工作台菜单点击后：

1. 读取当前 API 配置、环境、账户 profile
2. 把必要上下文传给套利专用进程
3. 启动独立进程
4. 主工作台可以保持打开，也可以最小化

这个方式最符合你的意思：

- 原程序还是统一入口
- 但这个工具启动后不再被主程序拖慢

### 5.2 独立资源

专用进程内：

- 独立 Tk root 或独立 UI 进程
- 独立 `OkxRestClient`
- 独立公共 WS
- 独立私有 WS
- 独立日志
- 独立内存状态

需要共享的只有：

- 凭证
- 基础配置
- 必要时的账本数据

现有项目已经支持指定独立数据目录，[main.py](/D:/qqokx/main.py:26) 和 [okx_quant/app_paths.py](/D:/qqokx/okx_quant/app_paths.py:24) 可以直接利用。

也就是说，极速版完全可以使用自己的数据根目录，例如：

- `qqokx_data/arbitrage_fast`

这样能避免和主工作台争用状态文件。

## 6. 运行模式重构

### 6.1 两段式模式

现货套利极速版建议只保留两个模式。

### 模式 A：准备模式

只做这些事：

- 读取当前持仓
- 选择当前交割合约
- 选择目标交割合约
- 确认移仓数量
- 预热需要的行情和账户状态

这阶段允许稍微慢一点，但要一次性完成。

### 模式 B：锁定交易模式

一旦点击“锁定合约组”，程序立即进入专用运行时：

- 停止一切全市场扫描
- 停止无关 tab 的刷新
- 停止无关图表
- 停止无关持仓轮询
- 只保留当前交易上下文

当前上下文通常只需要：

- 现货腿
- 当前交割合约腿
- 目标交割合约腿
- 账户和订单回报

这才是真正的“资源清场”。

## 7. 后台逻辑怎么按最高性能改

下面是我建议直接动刀的后台逻辑。

### 7.1 行情引擎改成单一热路径

不要再以“页面定时刷新”为核心。

改成：

- 独立 `QuoteEngine`
- 内部只维护当前锁定 instruments 的最新状态
- 公共 WS 作为主输入
- REST 仅冷启动和异常兜底

核心规则：

1. 锁定前可以订阅候选合约组
2. 锁定后只保留当前 2 到 4 个 instId
3. UI 只消费内存快照
4. 不再周期性主动全量拉盘口

### 7.2 只关心真正影响交易的字段

锁定交易后，很多通用行情信息都没必要反复算。

建议热路径只保留：

- bid1
- ask1
- last
- books5
- 盘口更新时间
- 当前可执行价差
- 当前分批剩余量

不在热路径上做：

- 全市场机会排序
- 历史图表刷新
- 大量格式化字符串拼接
- 多余的 UI 状态计算

### 7.3 执行链拆成专用 `RollExecutionEngine`

目前执行逻辑兼容开仓、平仓、移仓多场景。

极速版建议单独拆出：

- `RollExecutionEngine`

只为交割移仓服务。

专用执行链只处理：

1. 锁定当前腿和目标腿
2. 确认方向
3. 准备分批计划
4. 下当前合约回补单
5. 下目标合约开仓单
6. 通过私有 WS 追踪成交
7. 更新内存状态
8. 异步落盘

这样能减少大量条件分支。

### 7.4 私有 WS 升级为主确认通道

对极速交易来说，最关键的不是“界面刷新快”，而是：

- 下单后尽快确认是否成交
- 尽快知道是否要追单
- 尽快知道是否进入下一批

所以执行时必须把私有 WS 提到主路径。

执行确认链建议改成：

- 下单 REST
- 成交与订单状态确认主要靠私有 WS
- REST 查询订单状态只在超时补查时使用

### 7.5 本地账本改成内存主状态

当前 [okx_quant/arbitrage/position_ledger.py](/D:/qqokx/okx_quant/arbitrage/position_ledger.py:27) 是同步整份读写。

极速版建议改成：

- 进程启动时一次性装载账本
- 交易期间以内存对象为主
- 成交后异步刷新落盘
- 退出时强制 flush

必要时可以分成两个文件：

- `open_positions.json`
- `trade_journal.jsonl`

这样写入更轻。

### 7.6 日志改成低阻塞

极速版不要在热路径里频繁写大量格式化日志。

建议：

- UI 日志只保留关键节点
- 明细日志用异步队列
- 高频 quote 更新不写文件
- 只在状态切换时落日志

## 8. UI 也要配合“极限性能”

### 8.1 锁定后进入极简界面

锁定合约之后，UI 应该自动切到一个简版交易面板，只保留：

- 当前腿盘口
- 目标腿盘口
- 绝对价差
- 价差率
- 当前批次
- 执行按钮
- 自动移仓状态

其它内容都收起或移除：

- 机会扫描
- 套利图表
- 账本列表
- 冗长日志

### 8.2 BBO 优先

深度区不是第一优先。

首屏必须先给出：

- 买一
- 卖一
- 可执行价差
- 更新时间

深度档位可以延后补。

### 8.3 快捷键优先

如果是临战工具，建议引入快捷键：

- `F5` 刷新准备信息
- `Enter` 执行
- `Space` 启停自动移仓
- `1/2/3` 切换目标交割合约

这样比点 tab 更快。

## 9. 主程序如何保留入口

这里我建议不要做成完全脱离生态的另一个项目，而是做成：

- 同仓库
- 同打包体系
- 双入口

结构上可以是：

- 主工作台入口
- 套利极速版入口

主工作台内保留：

- “打开现货套利极速版”

点击后：

- 直接 `subprocess.Popen(...)` 启动专用进程

这样维护成本低，用户使用也统一。

## 10. 两种落地路径

### 路径 A：同仓库双入口

这是我最推荐的。

特点：

- 改造成本可控
- 代码仍可共享
- 便于发版
- 主程序可作为入口
- 独立进程性能收益大

### 路径 B：彻底拆成独立项目

特点：

- 隔离最彻底
- 但维护成本明显更高
- 配置同步、打包、升级、凭证共享都更麻烦

除非后面准备把套利做成完整产品，不然暂时不建议走这么远。

## 11. 我建议的实施顺序

### 第一阶段：先拆运行时

目标：

- 先让“现货套利”脱离主进程

动作：

- 新增独立启动入口
- 主菜单改成拉起子进程
- 独立 client / WS / 日志 / data root

这是收益最大的第一步。

### 第二阶段：再做锁定合约模式

目标：

- 让工具进入专用状态

动作：

- 新增“锁定合约组”
- 锁定后停掉无关模块
- 只保留当前上下文订阅

### 第三阶段：重写热路径执行链

目标：

- 把移仓执行压到最短路径

动作：

- 拆专用 `RollExecutionEngine`
- 私有 WS 主确认
- 内存账本 + 异步落盘

### 第四阶段：做临战 UI

目标：

- 真正做到“打开就能快交易”

动作：

- 极简面板
- 快捷键
- BBO 优先
- 状态高亮

## 12. 验收指标

如果按这条路做，我建议验收看这些指标。

### 启动指标

- 从主程序点击入口，到极速版可操作界面出现：`<= 2s`

### 锁定指标

- 锁定合约后，核心报价进入稳态：`<= 1s`

### 切换指标

- 在候选交割合约之间切换，首屏买一卖一更新：`<= 300ms`

### 执行指标

- 下单后首个有效订单状态回报，应主要由私有 WS 驱动

### 稳定性指标

- 锁定模式下不再出现无关扫描任务
- 热路径不再依赖整表文件读写

## 13. 最终建议

如果这次真的是按“最高性能”来定方向，我的建议是非常明确的：

- 不要继续把它当主工作台里的一个普通窗口修补
- 直接把“现货套利”升级成独立专用进程
- 锁定合约后进入极简专用运行时
- 后端按“只服务当前交易”重写热路径

这是最符合你业务场景的路线。

不是“在通用平台里把一个模块提速”，而是：

- “把一个临战工具从通用平台里解放出来”
