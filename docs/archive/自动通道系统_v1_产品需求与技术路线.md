# 自动通道系统 v1 产品需求说明与技术实施路线

## 1. 文档目的

本文档用于把“自动识别历史 K 线结构并绘制平行通道/箱体，再基于结构生成信号、回测并逐步接入实盘”的想法，整理为可执行的产品定义与开发路线。

这不是一次性大重构方案，而是基于当前 `D:\qqokx` 现有桌面交易系统进行小步升级的实施草案。

---

## 2. 产品定义

### 2.1 产品目标

构建一个面向 `OKX` 的桌面交易分析系统，使系统能够：

- 根据历史 K 线自动识别显著高低点
- 自动绘制上升通道、下降通道、横盘箱体
- 基于结构自动生成交易信号
- 使用同一套结构与信号逻辑进行回测评估
- 后续逐步接入模拟盘与实盘

### 2.2 产品本质

本产品不是简单的画线工具，也不是只做一个策略脚本。

本产品本质上是一个：

`自动结构识别 + 自动绘图 + 自动信号生成 + 回测评估 + 后续交易执行接入`

的一体化分析系统。

### 2.3 目标用户

- 需要基于结构交易的个人交易者
- 希望减少主观画线差异的半自动交易者
- 需要用统一规则验证“通道突破/回踩”策略是否有效的研究型用户

---

## 3. 核心需求

### 3.1 结构识别需求

系统需要基于历史 K 线自动识别：

- 显著高点
- 显著低点
- 上升通道
- 下降通道
- 横盘箱体
- 通道中轴线

### 3.2 通道识别核心逻辑

#### 上升通道

- 先识别显著低点
- 选择两个明显低点作为基准点
- 连接两个低点形成下轨支撑线
- 再从另一侧高点边界中寻找最匹配的平行线，形成上轨

#### 下降通道

- 先识别显著高点
- 选择两个明显高点作为基准点
- 连接两个高点形成上轨压力线
- 再从另一侧低点边界中寻找最匹配的平行线，形成下轨

#### 关键约束

- 基准线使用“显著高低点”
- 另一侧平行线不强制必须由同等级显著点构成
- 另一侧更强调“边界贴合能力”而非“枢轴显著性”

### 3.3 箱体识别逻辑

当结构满足以下条件时识别为箱体：

- 斜率接近 0
- 高低边界在一定误差范围内被重复测试
- 上下边界测试次数达到阈值
- 箱体宽度和持续时间满足最小要求

### 3.4 信号生成需求

系统需要支持以下结构信号：

- `BUY_BREAKOUT`
- `SELL_BREAKOUT`
- `BUY_RETEST`
- `SELL_RETEST`
- `SELL_AT_UPPER`
- `BUY_AT_LOWER`

其中 `v1` 第一优先只做：

- `BUY_BREAKOUT`
- `SELL_BREAKOUT`

### 3.5 图表展示需求

图表中需要展示：

- K 线
- 通道上下轨
- 箱体区域
- 中轴线
- 枢轴高低点
- 开仓信号
- 止盈止损线
- 当前结构说明

### 3.6 回测需求

系统需要使用历史数据回测结构策略，并输出：

- 胜率
- 盈亏比
- Profit Factor
- Sharpe
- 最大回撤
- 收益曲线
- 通道类型表现分析
- 交易记录导出

## 5. 产品规则定义 v1

### 5.1 枢轴点定义

枢轴点用于提供结构识别的候选锚点。

`v1` 建议支持三种方式，但先落一种稳定方案：

- `ZigZag`
- `Fractal Pivot`
- `ATR 波动过滤后的局部高低点`

建议实施顺序：

1. 先实现 `Fractal Pivot + ATR 过滤`
2. 再扩展 `ZigZag`
3. 后续支持多算法切换

### 5.2 显著点定义

显著点不是所有局部高低点，而是满足以下至少部分条件的点：

- 左右各有若干根 K 线确认
- 与前一个同类点相比有足够价格差
- 价格波动幅度超过 ATR 阈值
- 在时间上与前一关键点保持最小间距

### 5.3 通道生成定义

对每一段候选趋势结构：

1. 从显著点集合中寻找候选点对
2. 构造基准线
3. 计算所有另一侧价格点到该线的平行偏移量
4. 选择最能代表边界的偏移量形成平行线
5. 计算触碰次数、穿越次数、通道宽度稳定性
6. 对候选通道评分
7. 选取高分结构作为主结构

### 5.4 平行线生成原则

另一侧平行线不必再找两个同等显著点硬连，而应优先考虑：

- 最大边界包络合理性
- 触碰次数
- 穿透次数和穿透深度
- 与视觉边界的一致性

换句话说，另一侧线更像“平移拟合边界”，不是再次主观选点连线。

### 5.5 箱体定义

箱体识别的核心是：

- 上边界近似水平
- 下边界近似水平
- 区间震荡时间足够
- 顶底反复测试次数足够
- 区间内部不能只是单边趋势中的短暂停顿

## 6. v1 技术原则

### 6.1 架构原则

必须遵循以下原则：

- 小步迭代
- 保持现有项目始终可运行
- 优先复用现有模块
- 新能力模块化接入
- 回测和实盘尽量共用同一套分析与信号逻辑

### 6.2 与当前项目的关系

当前项目已经具备：

- 桌面 GUI 基础
- K 线数据获取与缓存
- 回测模块
- OKX 接口
- 策略运行框架

因此本项目不是从零做，而是做“二层架构升级”。

### 6.3 UI 技术选择

从长期能力看，`PySide6` 更适合复杂桌面平台。

但结合当前项目现状：

- 当前已有 `Tkinter + Canvas`
- 文档明确要求不能推翻重写
- 第一阶段重点是把内核能力跑通

因此 `v1` 建议：

- 先在现有 `Tkinter + Canvas` 上接入新结构识别和绘图
- 等分析内核稳定后，再评估 UI 层是否逐步迁移到 `PySide6`

这意味着：

`v1 优先做分析内核，不优先做 UI 技术迁移`

## 9. 分阶段开发清单

### 阶段 0：规则固化

目标：
先把规则定义清楚，避免后面开发时“边做边改口径”。

交付物：

- 显著点识别规则
- 通道识别规则
- 箱体识别规则
- Breakout 规则
- 若干样例图和人工标注结果

验收标准：

- 同一张图不同人理解结果大体一致
- 能明确“什么算识别成功”

### 阶段 1：结构识别内核

目标：
先让系统稳定输出结构识别结果。

开发项：

- `pivot_detector`
- `trendline_builder`
- `channel_detector`
- `box_detector`
- 结构评分机制

验收标准：

- 对固定历史数据能稳定输出主结构
- 输出结果可重复
- 有基础单元测试

### 阶段 2：图表展示

目标：
把识别结果直接画出来，能肉眼检查对不对。

开发项：

- K 线图叠加通道
- 箱体高亮
- 中轴线绘制
- Pivot 点标记
- 当前结构信息展示

验收标准：

- 打开历史行情能看到自动画线
- 用户能理解系统为何这么画

### 阶段 3：Breakout 信号

目标：
先做最清晰、最容易验证的一类信号。

开发项：

- 通道上破做多
- 通道下破做空
- 箱体上破/下破
- ATR 过滤
- 可选成交量过滤
- 假突破最小确认机制

验收标准：

- 图上能标出开仓信号
- 信号与识别结构一致

### 阶段 4：基础回测

目标：
验证结构策略是否有统计价值。

开发项：

- 历史 K 线回测
- 止盈止损规则
- 统计指标
- 报告导出

验收标准：

- 回测使用与图表展示一致的结构识别逻辑
- 每笔交易都能追溯到结构与信号来源

### 阶段 5：架构升级

目标：
在不破坏当前项目的前提下，为后续扩展打基础。

开发项：

- 新增 `EventBus`
- 分析结果事件化
- GUI 与分析层解耦
- 回测与 live 共用结构分析模块

验收标准：

- 模块间减少直接硬耦合
- 旧系统仍可运行

### 阶段 6：策略增强

目标：
从基础突破策略走向更完整的结构策略。

开发项：

- Retest
- Channel Reversal
- ATR 动态止损
- 通道失效止损
- RR 止盈
- 固定风险仓位

### 阶段 7：模拟盘与实盘接入

目标：
把结构分析系统与交易执行系统连起来。

开发项：

- 实时 K 线订阅
- 实时结构刷新
- 模拟盘
- OKX 实盘
- 仓位同步
- 风控监控

## 11. v1 关键风险

### 11.1 规则不稳定

最大的风险不是代码写不出来，而是规则太模糊，导致系统“乱画线”。

应对：

- 先固化样例和人工标准
- 先用固定数据回归验证

### 11.2 过早重构

如果一开始就改 GUI、重做引擎、改 OKX 执行链路，风险太高。

应对：

- 先做分析内核
- 再做展示
- 最后再接事件和执行

### 11.3 结构过拟合

自动通道很容易对样例图看起来正确，但换一段行情就失效。

应对：

- 结构评分要可解释
- 要跑多段历史回测
- 不只看单一图例

---

## 12. 当前推荐启动顺序

建议按以下顺序正式开始：

1. 先补一份“结构识别规则说明”
2. 选定 `v1` 的 Pivot 方案
3. 实现 `pivot_detector`
4. 实现 `channel_detector`
5. 实现 `box_detector`
6. 在现有图表上完成结构叠加显示
7. 再做 `Breakout`
8. 再接入回测

`把显著点、基准线、平行边界、箱体和信号的规则定义得足够清楚，并用小步迭代方式稳稳接入现有系统。`
