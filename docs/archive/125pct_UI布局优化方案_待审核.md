# 125% 字体缩放 UI 布局优化方案（待审核）

## 1. 背景

客户机器使用 Windows 125% 显示缩放后，回测窗口出现控件排布拥挤、按钮位置不稳定、说明文字挤压操作区的问题。  
当前现象不是 DPI 模糊，而是布局对更大字号和更小有效宽度适配不足。

本次先针对最明显的回测页面出优化方案，重点覆盖：

- 参数区底部操作行
- 图表区工具栏
- “图表大窗”按钮可见性与可点击性

## 2. 现状定位

### 2.1 回测窗口本身已开启 DPI aware

程序启动时已启用 Windows DPI aware，说明问题不在系统把整窗拉伸模糊，而在页面内部布局策略。

- 入口：`D:\qqokx\main.py`

### 2.2 回测页参数区是固定 6 列栅格

当前回测参数区域采用固定 6 列布局，很多“标签 + 输入框”都被强制塞在同一排。

- 参考：`D:\qqokx\okx_quant\backtest_ui.py:1774`

这类布局在 100% 下还能勉强成立，但在 125% 下会出现：

- 标签占宽变大
- 输入框可用宽度被压缩
- 说明文案、按钮组与输入区争抢同一行宽度

### 2.3 参数区底部把说明、同步按钮、开始回测塞在同一行

当前“回测K线数”这一行同时承载：

- 输入框
- 长说明文字
- “同步历史数据”
- “校验数据”
- “开始回测”

位置参考：

- `D:\qqokx\okx_quant\backtest_ui.py:2038`
- `D:\qqokx\okx_quant\backtest_ui.py:2044`

这是 125% 下最容易乱的区域之一。

### 2.4 图表工具栏把长说明和按钮塞在同一行

图表区工具栏当前是一行内同时放：

- 一段较长的交互说明
- “重置视图”
- “图表大窗”

位置参考：

- `D:\qqokx\okx_quant\backtest_ui.py:2460`
- `D:\qqokx\okx_quant\backtest_ui.py:2465`
- `D:\qqokx\okx_quant\backtest_ui.py:2468`

在 125% 下，这里会直接导致：

- 按钮被挤到最右边边缘
- “图表大窗”视觉上贴边甚至不易发现
- 长说明文字把按钮区顶得很紧

### 2.5 回测窗口初始宽度偏保守，且存在上限

当前回测窗口限制了宽度上限：

- `width_ratio=0.8`
- `min_width=1100`
- `max_width=1580`

参考：

- `D:\qqokx\okx_quant\backtest_ui.py:1348`

在高分屏 + 125% 缩放下，1580 的上限不算宽，容易让固定 6 列布局更早进入拥挤状态。

## 3. 优化原则

本次建议遵循以下原则：

1. 不靠“缩小字体”解决问题，优先改布局。
2. 不追求一次性大改页面结构，先做低风险、高收益调整。
3. 关键操作按钮必须与说明文字解耦，避免抢占同一行。
4. 在 100% / 125% / 150% 下都要保证主要按钮稳定可见。

## 4. 建议方案

## 方案 A：先做低风险响应式重排（推荐）

这是最适合先提交审核的方案，改动集中、风险较低、收益明显。

### A1. 图表工具栏拆成两行

当前问题：

- 说明文字和按钮共用一行

建议改法：

- 第 1 行只放说明文字，允许自动换行
- 第 2 行单独放操作按钮组
- 按钮组右对齐，保留“重置视图”“图表大窗”

预期效果：

- 125% 下“图表大窗”不会再被说明文字挤压
- 按钮点击区域更稳定
- 用户一眼能看到主操作入口

### A2. 参数区底部操作拆为“输入说明行 + 操作按钮行”

当前问题：

- “回测K线数”这一行承载内容过多

建议改法：

- 保留“回测K线数 + 输入框”
- 长说明文字独立成下一行
- “同步历史数据 / 校验数据 / 开始回测 / 当前参数单组回测”再独立成操作条

预期效果：

- 125% 下按钮不会和说明文字互相挤压
- 主操作区更清晰
- 用户更容易理解“先同步/校验，再回测”的顺序

### A3. 回测窗口默认宽度适当放宽

建议改法：

- 将回测窗口 `width_ratio` 从 `0.8` 提高到 `0.86` 或 `0.88`
- 将 `max_width` 从 `1580` 提高到 `1720` 或直接按屏幕可用宽度放开

预期效果：

- 在 2K 屏 + 125% 场景下，窗口更接近实际可用宽度
- 固定列布局的压力会明显下降

说明：

这一步只能缓解，不能替代响应式重排，因此必须和 A1、A2 搭配。

### A4. 长说明文字统一启用动态换行

建议改法：

- 对工具栏说明、报告摘要、提示语增加基于容器宽度的 `wraplength`
- 避免说明文本把按钮或输入框顶出布局

预期效果：

- 不同分辨率、不同缩放下，文本表现更稳定

## 方案 B：中期升级为“分组表单布局”

如果后续不仅是回测页，其他窗口也出现 125% 下拥挤问题，建议继续做这一层升级。

### B1. 参数区从固定 6 列改为分组块

建议把参数拆成 4 个组：

- 基础参数：策略、交易对、周期
- 指标参数：EMA、ATR、多周期过滤
- 风控参数：止盈止损、风险模式、手续费
- 时间与数据：开始时间、结束时间、回测K线数、历史数据操作

组内仍可使用 grid，但每组单独成块，不再要求整页所有字段横向严格对齐。

预期效果：

- 125% 下更稳定
- 新增参数时不容易破坏整体排版
- 用户扫描效率更高

### B2. 为窄宽度增加“自动降列”

建议：

- 宽窗口时保持双列/三列组块
- 宽度不足时自动降为单列堆叠

适用阈值可按窗口宽度判断，例如：

- `>= 1600`：宽屏布局
- `1300 - 1599`：两列分组
- `< 1300`：单列堆叠

这属于结构升级，收益大，但开发与回归成本也更高，建议放到第二阶段。

## 5. 本次建议优先落地内容

为了控制风险，建议本轮审核通过后先做以下 4 项：

1. 图表工具栏拆两行，确保“图表大窗”按钮稳定可见。
2. 参数区底部按钮区独立成单独操作行。
3. 回测窗口默认宽度放宽，降低 125% 下的拥挤概率。
4. 长说明文案统一支持动态换行。

这 4 项属于“低风险、可快速验证”的优化，不会改动业务逻辑。

## 6. 不建议的做法

以下方案不建议作为主方案：

- 单纯把字体改小
- 把按钮文案强行缩短成难理解的简称
- 继续维持同一行承载说明文字和多个主按钮
- 只调窗口宽度，不改布局

原因是这些做法要么影响可读性，要么只能暂时掩盖问题。

## 7. 验收标准

审核通过后建议按以下标准验收：

1. Windows 100% / 125% / 150% 下，回测窗口可正常打开且无主按钮遮挡。
2. “图表大窗”按钮在默认窗口宽度下始终可见、可点击。
3. “同步历史数据”“校验数据”“开始回测”不与长说明文字互相覆盖或挤压。
4. 参数区不出现明显错位、控件截断、按钮贴边。
5. 不影响现有回测、图表放大、历史数据同步逻辑。

## 8. 推荐结论

建议按“两阶段”推进：

- 第一阶段：先做方案 A，解决 125% 下最明显的可用性问题。
- 第二阶段：如后续仍有多页面缩放问题，再做方案 B，把固定 6 列表单升级为分组响应式布局。

如果只允许先做一轮小改，我建议优先处理“图表大窗按钮所在工具栏”和“回测参数底部操作行”，这两处最直观，也最容易让客户感知到优化效果。
