# 多周期策略开发对齐清单

来源方案：
`C:/Users/Windows/Downloads/qqokx 多周期策略升级方案.md`

目标：
在不影响现有单周期 EMA 动态策略的前提下，为 `qqokx` 增加“高周期趋势过滤 + 低周期入场”的多周期 EMA 动态策略，并保证回测与实盘逻辑一致、禁止未来函数、为后续三周期扩展留接口。

## 一、当前仓库的可复用基础

### 1. 单周期动态策略已成型

- `okx_quant/strategies/ema_dynamic.py`
  现有 `EmaDynamicOrderStrategy.evaluate(candles, config, ...)` 已完整承担低周期入场逻辑。
- 这部分适合继续作为“低周期入场内核”复用，不建议直接在原类里塞多周期条件，避免污染旧策略。

### 2. 回测层已经有“大周期映射小周期”的现成思路

- `okx_quant/backtest.py`
  已存在 `_build_cross_higher_tf_bias(...)`，用 `bisect_right` 按时间把高周期状态映射到低周期。
- 这说明仓库已经接受“不能按索引同步高低周期，只能按时间映射”的实现方式。
- 多周期动态策略应复用这个方向，但不要直接复用 `cross_higher_tf_*` 命名。

### 3. 参数系统已经是“按策略配置参数集”

- `okx_quant/strategy_parameters.py`
  现有做法不是所有策略共用同一组 UI 参数，而是通过 `StrategyParameterProfile` 决定每个策略显示哪些参数。
- 这非常适合多周期策略接入，因为我们可以只让新策略显示多周期专属参数，不污染旧策略界面。

### 4. 实盘与本地信号模式都已经围绕单策略循环运转

- `okx_quant/engine.py`
  现有动态策略实盘流程是：
  1. 拉取 `config.bar` 的 K 线
  2. 调用策略求信号
  3. 生成挂单/触发器
- 多周期接入后，实盘层的主要改造点不是风控，而是“同时拉取低周期和高周期，并形成统一上下文”。

## 二、对齐后的实现原则

### 1. 保持旧策略完全不变

- `EMA 动态委托-做多`
- `EMA 动态委托-做空`

以上两条策略的：
- 策略 ID
- 参数集合
- 回测结果
- 实盘行为

都不应发生变化。

### 2. 新策略单独落一套，不在旧类里打补丁

建议新增：

- `okx_quant/strategies/ema_dynamic_multi_timeframe.py`

建议新增策略 ID：

- `ema_dynamic_mtf_long`
- `ema_dynamic_mtf_short`

建议新增策略展示名：

- `EMA 动态委托-多周期多头`
- `EMA 动态委托-多周期空头`

### 3. 低周期负责入场，高周期只负责放行/拦截

V1 不改变低周期入场机制，继续沿用现有：

- EMA 动态委托
- ATR 止损/止盈
- 动态止盈
- 每波最多开仓次数

高周期只给出三态过滤结果：

- `long`
- `short`
- `neutral`

对应 V1 行为：

- 多头策略只在高周期 `long` 时允许新开多单
- 空头策略只在高周期 `short` 时允许新开空单
- 高周期反转后，默认“停止新增仓位”，不做强平

## 三、推荐的数据结构落点

### 1. 不建议复用 `cross_higher_tf_*` 字段

当前 `StrategyConfig` 已有：

- `cross_higher_tf_inst_id`
- `cross_higher_tf_bar`
- `cross_higher_tf_ref_ema_period`

这组字段是给 `EMA 突破/跌破` 系列准备的，语义是“更高周期参考 EMA 偏置”，不是通用多周期动态策略上下文。

如果直接复用，会带来两个问题：

- 命名语义不对，后续难维护
- 低周期动态策略与突破策略会被迫共用一套不完全相同的配置意义

### 2. 建议在 `StrategyConfig` 新增专用字段

建议新增：

```python
mtf_filter_inst_id: str | None = None
mtf_filter_bar: str | None = None
mtf_filter_fast_ema_period: int = 21
mtf_filter_slow_ema_period: int = 55
mtf_reversal_mode: str = "block_new_entries"
```

约定：

- `bar` 继续表示低周期
- `mtf_filter_bar` 表示高周期
- `mtf_filter_inst_id` 为空时默认等于 `inst_id`
- `mtf_reversal_mode` 在 V1 只支持：
  - `ignore`
  - `block_new_entries`

### 3. 建议补一层时间框架上下文抽象

建议新增：

```python
@dataclass(frozen=True)
class TimeframeContext:
    entry_inst_id: str
    entry_bar: str
    filter_inst_id: str
    filter_bar: str
    entry_candles: list[Candle]
    filter_candles: list[Candle]
```

建议放在：

- `okx_quant/strategies/ema_dynamic_multi_timeframe.py`
  或
- `okx_quant/models.py`

如果考虑后续三周期扩展，更推荐先放独立策略文件中，确认稳定后再公共化。

## 四、推荐的策略层设计

### 1. 旧策略保留原接口

保留：

```python
EmaDynamicOrderStrategy.evaluate(candles, config, ...)
```

不动它，确保旧逻辑零回归。

### 2. 新策略单独提供多周期接口

建议：

```python
class EmaDynamicMultiTimeframeStrategy:
    def evaluate(
        self,
        entry_candles: list[Candle],
        filter_candles: list[Candle],
        config: StrategyConfig,
        *,
        price_increment: Decimal | None = None,
    ) -> SignalDecision:
        ...
```

内部建议拆两步：

```python
def evaluate_filter_bias(filter_candles, config) -> str:
    ...

def evaluate_entry_signal(entry_candles, config, price_increment=None) -> SignalDecision:
    ...
```

实现上：

- 高周期只判断 `fast EMA > slow EMA` 或 `<`
- 低周期直接复用现有动态策略判断结果
- 最后做一次方向放行

这样好处是：

- 低周期逻辑不重写
- 未来扩三周期时，高周期过滤器可以继续独立演进

## 五、推荐的回测改造方式

### 1. 回测入口层先补“多周期载入”

当前 `run_backtest(...)` 只围绕单一 `config.bar` 拉取 `candles`。

多周期策略建议改成：

- 先拉低周期 `entry_candles`
- 再拉高周期 `filter_candles`
- 提前构建 `low_index -> high_bias` 或 `timestamp -> high_bias`

### 2. 必须按时间映射，不能按索引对齐

建议沿用当前高周期偏置构建思路：

```python
j = bisect.bisect_right(high_ts, low_candle.ts) - 1
```

映射规则必须是：

- 只允许使用 `<= low_candle.ts` 的最后一根已收盘高周期 K

这条是这次开发里最重要的正确性要求。

### 3. 推荐新增一段通用映射辅助函数

建议新增：

```python
def _build_mtf_filter_bias(
    entry_candles: list[Candle],
    filter_candles: list[Candle],
    fast_period: int,
    slow_period: int,
) -> list[str]:
    ...
```

返回：

- `long`
- `short`
- `neutral`

不要把多周期映射逻辑直接散落在 `_run_dynamic_backtest(...)` 主循环里。

### 4. 回测主循环建议保持“低周期推进”

建议模式：

1. 遍历低周期 K
2. 查询该时刻对应的高周期 bias
3. 运行低周期动态入场判断
4. 若方向允许，再开仓

这样能与实盘的“低周期触发，高周期过滤”一致。

## 六、推荐的实盘改造方式

### 1. 先抽一个统一的多周期取数入口

`engine.py` 当前动态策略分支很多，如果直接把高周期拉取逻辑分别塞进：

- 交易所托管动态策略
- 本地信号动态策略
- 观察模式

后面会很难维护。

建议先抽一个内部辅助函数，例如：

```python
def _load_dynamic_mtf_context(self, config: StrategyConfig) -> TimeframeContext:
    ...
```

职责：

- 拉低周期 candles
- 拉高周期 candles
- 都过滤成 `confirmed`
- 保证最小样本数

### 2. 高周期刷新频率要与低周期分离考虑

V1 可以先做“每轮都拉一次高周期”，先保证正确。

但代码结构上要预留后续优化位：

- 高周期缓存
- 仅在高周期收盘附近刷新

建议先把“取 K”和“评估策略”解耦，这样后面优化缓存不会牵扯下单逻辑。

### 3. 实盘反转处理 V1 只做“禁止新开仓”

如果高周期从 `long` 变成 `short`：

- 已存在持仓：继续交给原止盈止损逻辑管理
- 新信号：不再允许新增

不要在 V1 引入强平，这会让回测、实盘、持仓管理、通知文案一起变复杂。

## 七、参数系统与 UI 的落点

### 1. `strategy_parameters.py` 需要新增参数键

建议增加：

- `mtf_filter_bar`
- `mtf_filter_fast_ema_period`
- `mtf_filter_slow_ema_period`
- `mtf_reversal_mode`

如果后续支持跨品种过滤，再考虑加入：

- `mtf_filter_inst_id`

V1 同品种优先，可以先不暴露这个参数到 UI。

### 2. 新策略用独立参数画像

建议在 `STRATEGY_PARAMETER_PROFILES` 中新增：

- 多周期多头 profile
- 多周期空头 profile

并仅在新策略下显示：

- 低周期 `bar`
- 高周期 `mtf_filter_bar`
- 高周期快/慢 EMA
- 高周期反转处理

旧动态策略不显示这些字段。

### 3. Launcher 与 Backtest UI 都要同步

涉及：

- `okx_quant/ui_shell.py`
- `okx_quant/backtest_ui.py`

需要同步的不是只有控件，还包括：

- 参数草稿读写
- 模板序列化/反序列化
- 参数说明文案
- 策略切换时的显示/隐藏逻辑

## 八、导出与报告建议

`okx_quant/backtest_export.py` 建议补充以下信息：

- 低周期
- 高周期
- 高周期 EMA 快慢线参数
- 高周期反转处理模式
- 被过滤掉的入场信号数
- 高周期多头区成交次数
- 高周期空头区成交次数

建议这些统计先放在 `BacktestResult` 或辅助统计结构中，不要在导出阶段临时重算。

## 九、测试拆分建议

### 1. 策略层

文件建议：

- `tests/test_dynamic_strategy.py`
  新增多周期策略判断测试

必测：

- 高周期多头时允许低周期多头信号
- 高周期空头时拦截低周期多头信号
- 高周期中性时不放行

### 2. 回测层

文件建议：

- `tests/test_backtest.py`

必测：

- 高低周期按时间映射，不按索引
- 只读取已收盘高周期 K，不出现未来函数
- 高周期反转后停止新增，但不影响旧仓位出场
- 旧单周期回测行为不变

### 3. 实盘层

文件建议：

- `tests/test_strategy_engine.py`

必测：

- 实盘会同时拉取低周期/高周期
- 高周期不允许方向时不会生成新挂单/新触发器
- 新策略不会影响旧动态策略流程

## 十、推荐的开发顺序

### Phase 0：准备层

1. 新增策略 ID 与策略定义
2. 扩展 `StrategyConfig`
3. 扩展 `strategy_parameters.py`
4. 抽 `TimeframeContext` / 多周期 bias helper

### Phase 1：策略与回测

1. 新增 `ema_dynamic_multi_timeframe.py`
2. 在 `backtest.py` 接入新策略
3. 增补策略层与回测层测试

这是最优先阶段，必须先把“无未来函数 + 回测稳定”跑通。

### Phase 2：实盘

1. 抽统一多周期取数入口
2. 接入新策略到动态策略实盘分支
3. 增补 `test_strategy_engine.py`

### Phase 3：UI

1. Launcher 参数显示
2. Backtest 参数显示
3. 模板/草稿/序列化同步

### Phase 4：报告与版本信息

1. 回测导出增强
2. README / 软件开发指南更新
3. 版本号升级到 `v0.6.0`

## 十一、明确不在 V1 做的内容

以下内容本轮不建议进入首版：

- 三周期过滤
- 多过滤器叠加
- RSI / MACD
- 高周期反转强制平仓
- 跨品种高周期过滤
- 高频缓存优化到极致

V1 先做对：

- 结构隔离
- 回测正确
- 实盘一致
- 旧策略零回归

## 十二、开工前的最终落地结论

这次升级最合适的路径不是“改旧动态策略”，而是：

1. 在策略目录新增独立的多周期动态策略
2. 在 `StrategyConfig` 增加专用的多周期过滤字段
3. 在回测层先把“时间映射 + 无未来函数”做扎实
4. 再把同一套上下文结构接入实盘
5. 最后补 UI 和报告

如果按这个路径推进，能最大程度复用现有代码，同时把返工面控制在最低。
