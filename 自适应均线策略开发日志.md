# 自适应均线策略开发日志

日期：2026-05-13

## 一、策略理解

这套策略的核心，不是传统的 `EMA 金叉/死叉`，而是识别：

当前市场到底正在尊重哪一条 `EMA`，并把它当成推进轨道来运行。

策略目标不是预测，而是跟随已经形成的结构，围绕“被市场尊重的 EMA 轨道”反复交易，直到这条轨道第一次真正失效。

这意味着策略的关键，不在于均线本身，而在于下面几个定义是否严格：

- 什么叫“触碰轨道”
- 什么叫“回踩成功”
- 什么叫“假跌破”
- 什么叫“轨道失效”
- 什么叫“主导 EMA 切换”

如果这些定义不清晰，回测结果就会高度依赖主观解释，策略很容易出现“图上看着对，程序里做不稳”的问题。

---

## 二、总体判断

这个策略值得做，而且适合接入当前项目。

原因：

- 仓库里已经有成熟的 `EMA` / `ATR` 指标计算能力
- 已有回测主链路、策略目录、参数模型、风控骨架
- 现有结构支持新增独立策略，不需要从零搭系统

但这类策略最容易出问题的地方，不是代码实现，而是规则定义：

- 轨道识别是否会抖动
- 评分是否事后偏置
- 趋势过滤是否足够严格
- 同一段行情是否被重复计数

所以第一版最重要的目标不是“做复杂”，而是“做清楚”。

## 五、Respect Score 建议

第一版建议使用可解释的规则评分，不要直接上机器学习。

建议先做线性评分：

```python
score =
    touch_score
    + bounce_score
    + slope_score
    - fake_break_penalty
    - break_penalty
```

推荐评分维度：

1. 有效触碰次数
2. 回踩反弹成功率
3. EMA 斜率强度
4. 假跌破次数
5. 有效跌破次数

这样做的优势：

- 结果可解释
- 便于调参
- 回测时能知道到底哪项在贡献收益
- 后续替换为数据驱动权重也容易

## 六、最关键的定义建议

这部分是整个策略成败的核心。

### 1. touch 定义

建议不要用“价格接近 EMA”这种模糊说法。

建议定义为：

```python
abs(price - ema) <= k * ATR
```

例如：

```python
k = 0.2
```

也就是价格进入 `EMA ± 0.2 ATR` 区域，才算一次触碰。

### 2. bounce success 定义

建议定义为：

- 价格触碰 EMA 后
- 在后续 `N` 根 K 线内
- 至少走出 `Y * ATR` 的反弹幅度
- 且期间不能先发生有效跌破

这样可以避免把“碰一下就继续横盘”误判成成功支撑。

### 3. fake break 定义

建议定义为：

- 盘中跌破 EMA
- 但收盘重新站回
- 或下一根很快收回

这类情况不能直接算轨道失效，更适合作为“假突破/假跌破”计入惩罚或弱化项。

### 4. broken 定义

建议轨道失效不要只看单根影线。

建议优先用：

- 收盘有效跌破 `EMA - X * ATR`
- 连续若干根无法收回
- EMA slope 走平或转弱
- 高低点结构破坏

### 5. rail switch 定义

这是最容易抖动的地方。

建议不要出现 `EMA21` 和 `EMA34` 来回切换。

可以增加切换门槛：

- 新轨道分数必须高出旧轨道一个阈值
- 连续若干根都保持优势
- 或旧轨道先进入 `WEAKENING`

## 七、交易逻辑建议

### 1. 方向

第一版只做多，合理。

原因：

- `BTC` 长期偏多头
- 多头结构通常比空头更稳定
- 有利于先验证策略主干逻辑

### 2. 入场

文档里“前两次观察，第三次开始挂单”的思路很好，建议保留。

可以理解为：

- 前两次只用于确认该 EMA 的确被市场尊重
- 第三次开始参与，避免太早进场

建议入场区：

```python
entry_zone = EMA ± 0.2 ATR
```

### 3. 止损

建议沿用 ATR 动态止损，方向正确。

例如：

```python
stop = EMA - 1.2 ATR
```

但需要注意：

- 止损距离不能太近，否则轨道类策略会被正常回踩频繁洗掉
- 止损距离不能只凭经验，必须做矩阵回测

### 4. 止盈

不建议固定止盈，这一点认同。

轨道策略的优势本来就来自“让利润沿结构延伸”，固定止盈会切断大段趋势收益。

第一版建议：

- 以轨道失效作为主退出逻辑
- 必要时叠加保本上移机制

## 九、回测重点

这类策略回测不能只看胜率。

更重要的是看：

- 盈亏比
- 最大回撤
- 趋势行情贡献度
- 震荡行情损耗
- 连续亏损次数
- 不同市场阶段的表现分布

特别建议新增两类分析：

### 1. 轨道稳定性分析

记录：

- 主导 EMA 切换频率
- 单条轨道平均持续时长
- 每条轨道的平均触碰次数

如果轨道切换过于频繁，说明评分体系不稳定。

### 2. 信号质量分析

记录：

- 第 1 次、第 2 次、第 3 次回踩的胜率差异
- 不同 EMA 周期的胜率和盈亏比
- 不同 `ATR` 宽度下的表现

这样才能知道：

是“所有轨道都有效”，还是“只有某几类轨道有效”。

## 十八、市场状态过滤方案

V1 建议不要一口气上太多过滤器。

推荐第一版只做三项：

1. `价格收盘在 EMA200 上方`
2. `EMA200 slope > 0`
3. `近 N 根高点和低点整体抬高`

可以先不接 `ADX`。

原因：

- 你当前指标层里还没有现成 `ADX`
- 先用简单趋势判定，可以更快进入验证阶段
- 后续如果震荡过滤不够，再补 `ADX` 或波动过滤

建议输出：

```python
TREND_UP
RANGE
UNQUALIFIED
```

V1 不需要把 `TREND_DOWN` 和 `VOLATILE` 做得很复杂，因为当前只做多。

### 建议判定口径

```python
trend_ok =
    close > ema200
    and ema200_now > ema200_n_bars_ago
    and recent_swing_highs_lows_rising
```

其中：

- `ema200_now > ema200_n_bars_ago` 用于判断大方向抬升
- `recent_swing_highs_lows_rising` 用于避免 EMA 刚抬头但结构还没站稳

建议每根新 K 线都对候选 EMA 重新评分，但不要每次都立刻切换主轨道。

### 候选集合

```python
(5, 8, 13, 21, 34, 55, 89, 144, 233)
```

### 评估窗口

```python
lookback = 60 bars
```

### 对每条 EMA 计算的中间量

- `touch_count`
- `bounce_count`
- `bounce_rate`
- `fake_break_count`
- `hard_break_count`
- `slope_strength`
- `last_touch_age`

### slope_strength 建议

不要直接用价格差，而要归一化到 ATR。

建议：

```python

```

优点：

- 不同价格区间可比
- 后续用于分数加权更稳定

## 二十、Respect Score 初版公式

建议第一版用 100 分制，保持可读性。

### 评分公式

```python
respect_score =
    touch_score
    + bounce_score
    + slope_score
    + recency_score
    - fake_break_penalty
    - hard_break_penalty
```

### 推荐拆分

```python
touch_score = min(touch_count, 5) * 8
bounce_score = min(bounce_count, 4) * 12
slope_score = clamp(slope_strength, 0, 2) * 10
recency_score = max(0, 10 - last_touch_age)
fake_break_penalty = fake_break_count * 6
hard_break_penalty = hard_break_count * 12
```

### 评分解释

- `touch_score` 代表市场是否多次和该 EMA 互动
- `bounce_score` 代表这些互动是否真的产生了支撑效果
- `slope_score` 代表轨道是否具有推进方向
- `recency_score` 防止系统长期抱着一条“老轨道”不放
- `fake_break_penalty` 和 `hard_break_penalty` 用来打击脆弱轨道

### V1 主轨道选择规则

```python
best_ema = argmax(respect_score)
```

但只有在以下条件同时满足时才认定为候选主轨道：

- `touch_count >= rail_min_touches`
- `bounce_count >= rail_min_bounces`
- `slope_strength > 0`
- `respect_score >= 55`

## 二十一、touch / bounce / break 的程序定义

这一节必须写严，否则代码和肉眼会对不上。

### 1. touch

建议定义：

```python
abs(low_or_close - ema_value) <= rail_touch_atr_ratio * atr_value
```

对于多头支撑轨道，更推荐使用：

```python
low <= ema + zone and close >= ema - zone
```

这样既承认价格回踩过，又避免把完全跌穿后的远离位置误当成 touch。

### 2. bounce success

建议定义：

- 发生 touch 后
- 后续 `rail_bounce_confirm_bars` 根内
- 最高价相对 touch bar 至少上行 `rail_bounce_atr_ratio * ATR`
- 且期间没有先发生 hard break

### 3. fake break

建议定义：

- 盘中跌到 `EMA - touch_zone` 下方
- 但收盘重新站回 `EMA`
- 或下一根 K 线重新站回 `EMA`

### 4. hard break

建议定义：

```python
close < ema - rail_break_atr_ratio * atr
```

或者：

- 连续 `rail_reclaim_bars` 根都未能收回 EMA

V1 可以把两种都纳入 hard break。

## 二十二、状态机落地建议

文档里原始状态很多，理念很好，但第一版代码建议先简化成 5 个运行态。

### 建议的运行态

```python
SEARCHING
CANDIDATE
CONFIRMED
ACTIVE
BROKEN
```

### 状态说明

`SEARCHING`

- 正在趋势市场里寻找可能的主导 EMA

`CANDIDATE`

- 已找到当前最高分 EMA
- 但还没达到最小确认条件

`CONFIRMED`

- 已满足触碰、反弹、方向一致等条件
- 允许等待下一次回踩入场

`ACTIVE`

- 已存在该轨道上的交易行为
- 轨道继续有效时允许反复参与

`BROKEN`

- 当前轨道被确认破坏
- 进入冷却和重新寻找阶段

### 建议状态转移

```python
SEARCHING -> CANDIDATE
当某条 EMA 分数最高且超过最低阈值

CANDIDATE -> CONFIRMED
当该 EMA 满足最少 touches / bounces / slope

CONFIRMED -> ACTIVE
当价格再次进入 entry zone 并触发交易

ACTIVE -> BROKEN
当发生 hard break 或趋势过滤失效

BROKEN -> SEARCHING
经过冷却 bars 后重新评估
```

### 为什么要简化

因为第一版真正影响收益的不是状态名字，而是：

- 什么时候允许交易
- 什么时候禁止交易
- 什么时候认定轨道废了

这些明确了，之后再映射回更细的展示状态也不晚。

### 入场

建议沿用你文档里的“前两次确认、第三次参与”。

可程序化为：

- 历史上已出现至少 `2` 次成功支撑

## 二十九、开发风险提醒

这套策略最容易踩的坑有四个。

### 1. 未来函数

`bounce success` 的确认一定不能偷看未来太多 bars。

解决建议：

- 识别层和交易层严格区分
- 回测里只允许使用当时已经可见的信息

### 2. 轨道切换抖动

如果没有切换阈值，`EMA21` 和 `EMA34` 很容易来回抢主导地位。

解决建议：

- 引入 `rail_switch_min_score_delta`
- 引入最短持有轨道 bars

### 3. 评分过拟合

如果参数太多，回测很好看，但出样本会塌。

解决建议：

- V1 限制可调参数数量
- 先做少量默认值回测，不急着扫参

### 4. 震荡误判

趋势过滤不够时，轨道策略在横盘里会被来回打脸。

解决建议：

- 先用严格的 `EMA200 + 结构抬高` 过滤
- 后续再决定是否补 `ADX`
