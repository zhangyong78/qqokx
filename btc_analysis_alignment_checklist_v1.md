## 1. 项目定位

### 1.1 当前确认
- 这是一个新的 `BTC 自动技术分析系统` 需求
- 目标不是单一指标工具，而是完整分析链路：
  - 数据
  - 概率
  - 指标
  - K 线形态
  - 结构形态
  - 评分
  - 多周期共振
  - JSON 输出
  - 推送

### 1.2 需和领导确认
- 该系统是：
  - 作为当前 `OKX 策略工作台` 的一个新模块
  - 还是独立子系统
- 第一阶段是否只做 `BTC-USDT-SWAP`
- 第一阶段是否只做 `现货/永续` 其中一种，还是都支持

### 1.3 建议值
- 建议先作为当前仓库内的独立子系统开发，不直接混入现有交易策略代码
- 建议第一阶段只支持 `BTC-USDT-SWAP`
- 建议第一阶段只分析，不自动下单

## 4. 输出 JSON 口径

### 4.1 当前确认
- 所有输出必须为 JSON
- 必须有 `reason`
- 最终目标示例：

```json
{
  "direction": "long",
  "score": 8,
  "confidence": 0.7,
  "signals": [],
  "reason": []
}
```

### 4.2 需和领导确认
- `direction` 是否允许：
  - `long`
  - `short`
  - `neutral`
- `confidence` 取值范围是否固定为 `0 ~ 1`
- `signals` 中每条信号是否需要包含：
  - `name`
  - `category`
  - `score`
  - `strength`
  - `trend_context`
- `reason` 是否要求中文可读

### 4.3 建议值
- 建议统一为：

```json
{
  "symbol": "BTC-USDT-SWAP",
  "timeframe": "1H",
  "direction": "long",
  "score": 8,
  "confidence": 0.70,
  "signals": [
    {
      "name": "ema_bullish_alignment",
      "category": "indicator",
      "score": 2,
      "strength": 0.8,
      "trend_context": "uptrend"
    }
  ],
  "reason": [
    "EMA 多头排列",
    "MACD 金叉",
    "1H 与 4H 方向一致"
  ]
}
```

## 5. 概率分析

### 5.1 当前确认
- 概率分析是最高优先级
- 包含：
  - 单 K 线延续概率
  - 连续 K 线概率
  - 概率衰减

### 5.2 需和领导确认
- “概率”是基于什么统计样本：
  - 固定历史窗口
  - 全历史
  - 滚动窗口
- 连续 K 线概率是只看涨跌方向，还是要细分：
  - 连阳连阴
  - 实体大小
  - 上下影长度
  - 波动率环境
- 概率衰减如何定义：
  - 连续上涨后延续概率降低
  - 还是某种信号在时间上失效

### 5.3 建议值
- 第一阶段先做滚动窗口统计
- 先按方向级别实现：
  - 单阳后继续上涨概率
  - 单阴后继续下跌概率
  - 连阳 N 根后继续上涨概率
  - 连阴 N 根后继续下跌概率
- 概率衰减先定义为：
  - 连续同向 K 线数量增加后，延续概率的边际变化

## 6. 指标分析

### 6.1 当前确认
- 指标范围：
  - EMA
  - MACD
  - BOLL
- EMA 需要：
  - 乖离
  - 斜率
  - 排列

### 6.2 需和领导确认
- MACD 参数是否固定使用常用参数
- BOLL 参数是否固定使用常用参数
- EMA 周期是否固定，还是可配置

### 6.3 建议值
- 第一阶段默认参数：
  - EMA：`21 / 55 / 233`
  - MACD：`12 / 26 / 9`
  - BOLL：`20 / 2`
- 输出中保留参数字段，便于后续调整

## 8. 结构形态

### 8.1 当前确认
- 需要：
  - 双顶 / 双底
  - 趋势线突破

### 8.2 需和领导确认
- 双顶双底是用“收盘价结构”还是“高低点结构”
- 趋势线突破是否接受“近似算法”，还是要可视化级别严格拟合

### 8.3 建议值
- 第一阶段先用可解释、规则型算法：
  - 局部高低点识别
  - 简化趋势线突破
- 不做复杂机器学习识别

## 9. 评分系统

### 9.1 当前确认
- 评分系统是核心功能
- 需求示例：
  - EMA 多头 `+2`
  - MACD 金叉 `+2`
  - 突破 `+3`

### 9.2 需和领导确认
- 是否需要固定权重表
- 是否允许后续通过配置调整权重
- 最终 `direction` 如何由 `score` 映射得出

### 9.3 建议值
- 第一阶段采用固定权重表
- 评分拆三层：
  - `probability_score`
  - `indicator_score`
  - `pattern_score`
- 最终综合为 `total_score`

建议初版映射：
- `score >= 7`：强多 / 强空
- `score 4 ~ 6`：偏多 / 偏空
- `score <= 3`：中性或弱信号

### 10.1 当前确认

### 10.2 需和领导确认
- 是否必须 `3/3` 一致才算强信号
- `周线` 是否纳入第一阶段

### 10.3 建议值
- 第一阶段分析：
  - `1H`
  - `4H`
  - `1D`
- 共振规则建议：
  - `3/3` 一致：强信号
  - `2/3` 一致：中信号
  - `1/3` 或冲突：弱信号 / 中性

- 标的：`BTC-USDT-SWAP`

## 15. 我方建议的对齐结论

如果需要我给一个最稳妥、最容易落地的建议版本，建议先这样定：

- 项目形态：当前仓库内新增独立分析子系统
- 第一阶段范围：`BTC-USDT-SWAP + 1H/4H/1D + 概率分析 + JSON + 邮件`
- 第二阶段范围：指标、K 线形态、结构形态
- 第三阶段范围：评分系统、多周期共振、综合决策
- 第四阶段范围：调度、企业微信、QQ

这样好处是：
- 可以快速有第一个可演示版本
- 不会一开始就把范围铺太大
- 后续每一阶段都能拿结果给领导确认
