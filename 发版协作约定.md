# 发版协作约定

这个文件用于解决一个固定问题：

- 代码可能分散在不同线程里完成
- 最后打包、改版本号、写 `README.md` 更新日志，统一交给 Codex 收口
- 不再依赖某一个聊天窗口记住 `123` 的口头约定

## 1. 统一规则

- 功能线程里说 `123`
  - 默认表示：把当前线程已经完成、准备进入下一版的内容登记到 `发版待打包清单.md`
  - 默认不直接打包
  - 默认不直接提交和推送
- `总控-版本路线图` 或 `数据-缓存-导出-发布` 线程里说 `123`
  - 默认表示：读取 `发版待打包清单.md`
  - 统一判断版本号
  - 更新 `pyproject.toml`
  - 更新 `okx_quant/__init__.py`
  - 更新 `README.md` 当前版本与更新日志
  - 执行打包
  - 提交
  - 推送

## 2. 功能线程需要登记什么

每次只登记已经完成、准备进入下一版的结果，不写具体实现细节。

建议登记：

- 线程名
- 模块或主题
- 面向用户的变化摘要
- 版本建议：`patch` / `minor` / `major`
- 备注：是否暂时只登记、不发版

## 3. 版本号默认规则

- `patch`
  - 小修复
  - UI 可读性调整
  - 文案、交互、展示层修正
- `minor`
  - 新增功能
  - 明显增强
  - 用户可感知的新能力
- `major`
  - 破坏兼容
  - 大范围重构并改变原有行为

如果清单里没有特别说明，默认按 `patch` 递增。

## 4. 为什么要这样做

- Codex 能稳定读取仓库文件，但不能读取别的聊天窗口里的隐含记忆
- 把变更记到仓库里，任何线程都能查到同一份记录
- 发版线程只需要消费清单，不需要重新猜每个线程做了什么

## 5. 使用建议

- 功能线程结束前，让 Codex 先登记一条
- 总控线程发版前，先检查工作区有没有无关脏改
- 还没确认完成的内容，不要提前登记到发版清单
- 如果只想记录，不想立刻发版，可以明确说：只登记，不打包

## 6. 旧线程的一次性启用方式

如果某个线程是在这套约定落地前就已经开的，直接发裸 `123` 仍然可能被理解成别的操作。

这种旧线程第一次建议补一句：

```text
本线程启用 123 约定：我说 123 时，只登记当前线程已完成内容到 发版待打包清单.md，不做别的。
```

说过一次之后，这个线程后面继续发 `123` 就会稳很多。
