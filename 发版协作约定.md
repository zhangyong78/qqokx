# 发版协作约定

这份文档用来解决一个固定问题：

- 代码可能分散在不同线程里完成
- 最后由总控线程统一做版本号、更新日志、打包、提交和推送
- 不再依赖某个聊天窗口里的口头约定来决定是否发版

## 1. 统一规则

- 功能线程里说 `123`
  - 默认表示：当前线程已经完成本轮内容，准备进入下一版的发版收口
  - 默认动作：把这轮内容登记到 `发版待打包清单.md`
  - 默认不表示：只做本地保存、不发版
- 总控线程里说 `123`
  - 默认表示：读取 `发版待打包清单.md`
  - 统一判断版本号
  - 更新 `pyproject.toml`
  - 更新 `okx_quant/__init__.py`
  - 更新 `README.md` 的当前版本和中文更新日志
  - 更新 `软件开发指南.md`
  - 执行打包
  - 提交
  - 推送

## 2. 功能线程应该登记什么

功能线程只登记“已经完成、准备进入下一版”的结果，不写具体实现细节。

建议登记内容：

- 线程名称
- 模块或功能名
- 面向用户的变化摘要
- 版本建议：`patch` / `minor` / `major`
- 备注：是否只是暂存、不立刻发版

建议不要登记：

- 大段代码实现
- 调参过程
- 临时探索结论

## 3. 总控线程的职责

总控线程要负责最后的收口动作：

- 汇总各功能线程的完成内容
- 统一版本号
- 统一更新日志
- 执行打包
- 提交并推送到 GitHub

如果有冲突或不确定项，总控线程要先把边界说清楚，再继续发版。

## 4. 版本号规则

- 默认按 `patch` 递增
- 当用户明确说要切版本线，按新的版本线继续
- 版本号必须和以下文件保持一致：
  - `pyproject.toml`
  - `okx_quant/__init__.py`
  - `README.md`
  - `软件开发指南.md`
  - `发版待打包清单.md`

### 4.1 版本递增判断

- `patch`
  - 修文档
  - 修提示文案
  - 修 UI 小问题
  - 修测试
  - 修 bug
  - 不改变主要功能结构
- `minor`
  - 新增一个明显的新功能
  - 新增一个新窗口、新面板、新工作流
  - 原有功能能力明显增强
  - 但没有把整条产品线推翻重来
- `major`
  - 版本线切换
  - 架构大改
  - 数据目录、发版链路、线程协作规则这种底层约定大重构
  - 影响面很大，后续兼容要专门处理

### 4.2 判断原则

- 以“用户可感知的最大变化”作为版本级别判断依据
- 如果同一版里既有小修小补，又有大功能，优先按影响最大的变化判断
- 如果只是记录、整理、润色和文档收口，通常按 `patch`

## 5. 使用建议

- 功能线程结束时，先登记，再等总控线程发版
- 总控线程发版前，先检查工作区里有没有临时文件、缓存、报告产物
- 如果只想登记不发版，可以明确说“只登记，不发版”
- 如果要发版，直接说 `123`

## 6. 一次性启用写法

如果某条线程已经在这套规则之前就存在，可以补一句：

```text
本线程启用 123 约定：
我说 123 时，只表示把当前线程已经完成的内容登记到发版清单，
由总控线程统一决定是否进入下一版发版流程。
```

这样后续再说 `123`，含义会更稳定。
